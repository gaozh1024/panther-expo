## 二、经讨论确认的最优架构方案

### 1. Monorepo结构优化（架构师A主导）

```
panther-expo/
├── packages/
│   ├── core/               # 核心基础设施
│   ├── theme/              # 主题系统（纯配置，零依赖）
│   ├── ui/                 # UI组件库（依赖theme）
│   ├── utils/              # 工具函数
│   └── config/             # 【新增】共享配置
│       ├── tsconfig/
│       ├── eslint/
│       └── jest/
├── apps/
│   ├── test-app/           # 测试应用（从examples迁移）
│   └── playground/         # 【新增】组件开发调试
├── templates/
│   └── rn-starter/
└── tooling/
    ├── scripts/            # 构建脚本
    └── test-utils/         # 测试工具
```

**关键决策**：保持单向依赖链

```
utils ← theme ← core ← ui
```

---

### 2. 包导出重构（架构师B主导）

将UI包从单入口改为细粒度导出：

```json
// packages/ui/package.json
{
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",
      "import": "./dist/index.mjs",
      "require": "./dist/index.cjs"
    },
    "./button": {
      "types": "./dist/components/button/index.d.ts",
      "import": "./dist/components/button/index.mjs"
    },
    "./input": {
      "types": "./dist/components/input/index.d.ts",
      "import": "./dist/components/input/index.mjs"
    }
    // ... 每个组件独立导出
  }
}
```

**性能目标**：
| 指标 | 当前 | 目标 |
|------|------|------|
| UI包体积 | ~500KB+ | <200KB (按需加载) |
| Tree-shaking率 | ~30% | >80% |

---

### 3. 构建系统升级（架构师B主导）

**迁移到tsup**：

```json
// packages/*/package.json
{
  "scripts": {
    "build": "tsup src/index.ts --format cjs,esm --dts --treeshake --splitting"
  }
}
```

**pnpm优化配置**：

```bash
# .npmrc
shamefully-hoist=true
strict-peer-dependencies=false
auto-install-peers=true
```

---

### 4. 测试策略（架构师C主导）

```
测试架构：
├── packages/
│   └── */src/
│       └── *.test.ts          # 单元测试 (Vitest)
├── apps/test-app/
│   └── e2e/                   # E2E测试 (Maestro)
└── tooling/test-utils/
    └── src/
        ├── render.tsx         # RN测试渲染器
        └── mocks/             # 共享mock
```

**测试工具链**：

- **单元测试**: Vitest + @testing-library/react-native
- **E2E测试**: Maestro（比Detox更易配置）
- **覆盖率**: v8 provider，阈值80%

| **文档**     | TypeDoc自动生成API文档               |
| **调试**     | playground应用用于组件开发           |
| **模板**     | rn-starter改为依赖@panther-expo/\*包 |

---

### 6. 主题系统统一（架构师A+B共识）

**决策**：采用**分层主题架构**

```typescript
// theme包：纯设计令牌
export const tokens = {
  colors: { primary: '#f38b32', ... },
  spacing: { sm: 4, md: 8, ... },
  // ...
};

// ui包：运行时主题
import { tokens } from '@panther-expo/theme';
export function GluestackUIProvider({ theme, children }) {
  // 使用tokens + 运行时主题覆盖
}
```

**优势**：

- theme包保持纯净，可被任意项目使用
- ui包负责运行时动态主题切换
- 避免两个ThemeProvider冲突

## 四、关键架构决策记录（ADR）

### ADR-1: 构建工具选择

**决策**: 使用 `tsup` 替代 `tsc`

- **理由**: 支持双格式输出、内置tree-shaking、更快的构建速度
- **反对意见**: tsc更稳定
- **最终决定**: tsup基于esbuild，社区广泛采用，风险可控

### ADR-2: E2E测试工具

**决策**: 使用 `Maestro` 替代 `Detox`

- **理由**: 配置更简单、Expo生态支持更好
- **权衡**: Detox更底层，但配置复杂度高

### ADR-3: 主题系统架构

**决策**: 分层架构（theme设计令牌 + ui运行时主题）

- **理由**: 避免双轨制冲突，保持theme包纯净
- **替代方案**: 完全迁移到Gluestack主题系统（被否决，丧失灵活性）

### ADR-4: 单元测试框架

**决策**: 使用 `Vitest` 替代 `Jest`

- **理由**: 与Vite生态一致、更快的测试速度、更好的TypeScript支持
- **注意**: 需要配置react-native测试环境

## 五、立即执行清单

如果今天开始实施，建议按以下顺序：

```bash
# Day 1: 基础配置
1. pnpm add -D tsup -w
2. 更新所有包的package.json，添加子路径导出
3. 明确添加包间依赖（ui依赖theme）

# Day 2-3: 测试框架
4. pnpm add -D vitest @testing-library/react-native -w
5. 配置vitest.config.ts和测试环境
6. 编写第一个测试用例

# Day 4-5: 工程化
7. npx husky-init && pnpm install
8. 配置ESLint + Prettier
9. 添加lint-staged

# Week 2: 性能优化
10. 迁移所有包到tsup构建
11. 添加包大小监控
12. 重构模板项目
```
